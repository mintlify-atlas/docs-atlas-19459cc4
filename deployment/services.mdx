Launchpad is composed of multiple long-running daemons, Celery workers, and cron scripts. In development, most of these are started alongside the main appserver by `bin/run`. In production they run as separate system services.

## Starting Services in Development

The `bin/run` script accepts a `-r` flag to launch auxiliary services:

```bash
# Start with Librarian, Bing stub, Memcached, and RabbitMQ
bin/run -r librarian,bing-webservice,memcached,rabbitmq -i development

# Start everything including SFTP and code browsing
bin/run -r librarian,sftp,codebrowse,bing-webservice,memcached,rabbitmq -i development

# Start only the code-hosting stack
bin/run -r librarian,sftp,codebrowse,rabbitmq -i development
```

Each service name passed to `-r` corresponds to a `launch: True` entry in the config (e.g. `[librarian_server] launch: True`) or a named service registered in the run script.

## Twisted Daemons

All long-running Launchpad daemons are implemented as Twisted applications using `.tac` (Twisted Application Configuration) files in the `daemons/` directory. They are launched with `twistd`:

```bash
bin/twistd -noy daemons/<name>.tac
```

    **The Build Farm Dispatcher** — Scans registered build workers (`BuilddManager`), dispatches pending build jobs, monitors build progress, and processes upload results. ## The Librarian in Detail

The Librarian is Launchpad's central binary file store, used for build logs, package files, translation tarballs, bug attachments, and more. It is separate from the main database to allow large files to be served efficiently.

```ini
[librarian]
upload_port: 58090
download_port: 58080
download_url: http://launchpad.test:58080/
restricted_upload_port: 58095
restricted_download_port: 58085
restricted_download_url: http://launchpad.test:58085/
use_https: False

[librarian_server]
root: /var/tmp/fatsam
launch: True
logfile: librarian.log
```

Files stored through the restricted interface have the `restricted` flag set and are only accessible to authenticated users. In production, the Librarian stores files in OpenStack Swift; in development, files go to the local filesystem under `librarian_server.root`.

To start or stop the Librarian independently:

```bash
bin/start_librarian
bin/killservice librarian
```

## Celery Workers and Job Queues

Launchpad uses [Celery](https://docs.celeryq.dev/) for asynchronous background jobs. Jobs are dispatched via RabbitMQ and consumed by Celery workers. The queues and their configuration are defined in `configs/schema-lazr.conf`:

| Queue | Timeout | Concurrency | Purpose |
|---|---|---|---|
| `launchpad_job` | 300s | 3 | Main job queue |
| `launchpad_job_slow` | 86400s | 1 | Overflow for timed-out main jobs |
| `bzrsyncd_job` | 300s | 3 | Bazaar sync jobs |
| `bzrsyncd_job_slow` | 86400s | 1 | Overflow for timed-out Bazaar jobs |
| `branch_write_job` | 300s | 3 | Branch write operations |
| `branch_write_job_slow` | 86400s | 1 | Overflow for timed-out branch writes |
| `celerybeat` | 86400s | 1 | `RunMissingReady` periodic scheduler |
| `native_publisher_job` | 300s | 3 | Native package publishing |
| `native_publisher_job_slow` | 86400s | 1 | Overflow for publishing jobs |

Each queue has an optional `fallback_queue`: if a job exceeds the `timeout` limit, it is re-queued in the fallback queue. `launchpad_job_slow` and similar queues have no fallback and a 24-hour timeout.

Job sources (the Celery task classes) are listed in `[process-job-source-groups].job_sources` and include entries like:

```ini
[IBranchModifiedMailJobSource]
module: lp.code.interfaces.branchjob
dbuser: send-branch-mail
crontab_group: MAIN
```

Jobs in the `MAIN` crontab group are dispatched by `cronscripts/process-job-source-groups.py`. Jobs with `runner_class: TwistedJobRunner` (e.g. `IBranchMergeProposalJobSource`) are run in the Twisted event loop.

## RabbitMQ

RabbitMQ is the AMQP message broker used by Celery and for OOPS (error report) delivery. In development, `bin/run -r rabbitmq` starts a local RabbitMQ instance:

```ini
[rabbitmq]
launch: True
broker_urls: amqp://guest:guest@localhost:56720//
```

The `broker_urls` key accepts a space-separated list of AMQP URLs for round-robin failover across brokers.

OOPS reports are published to the `error_exchange` AMQP exchange:

```ini
[error_reports]
oops_prefix: X
error_dir: /var/tmp/lperr
error_exchange: none   # set to 'oopses' in production
```

## Memcache

Launchpad caches frequently read, rarely changed data (e.g. rendered branch listings, user karma) in Memcached. In development, a local instance is started automatically:

```ini
[memcache]
servers: (127.0.0.1:11217,1)

[memcached]
launch: True
address: 127.0.0.1
port: 11217
memory_size: 2
```

The `servers` format is `(host:port,weight)` — the weight value after the comma was formerly a debug flag and is now ignored.

## Code Browsing (Loggerhead)

Launchpad's Bazaar code browser (Loggerhead) runs as a separate service:

```bash
# Run in foreground
make run_codebrowse

# Run as daemon
make start_codebrowse

# Stop
make stop_codebrowse
```

Loggerhead is started via `scripts/start-loggerhead.py` with `BRZ_PLUGIN_PATH=brzplugins` set in the environment. It is configured in the `[codebrowse]` section:

```ini
[codebrowse]
cachepath: /var/tmp/bazaar.launchpad.test/cache
log_folder: /var/tmp/bazaar.launchpad.test/logs
launchpad_root: https://code.launchpad.test/
secret_path: configs/development/codebrowse-secret
```
